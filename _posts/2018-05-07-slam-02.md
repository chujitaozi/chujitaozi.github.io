# 2.1 引子

SLAM主要应用还是在未知环境，所以主要依靠携带在机器人上的传感器来进行间接地测量，不能使用导轨、二维码等在环境中的传感器。视觉SLAM，主要就是指如何使用相机解决定位和建图的问题。

相机并不是指单反，而是能够以一定频率拍摄周围环境的，能形成视频流的摄像头。一般来说，普通的摄像头能够以30帧/s采集图像。按照工作方式，相机可以分为单目相机、双目相机和深度相机三大类。从名称就能看出来，单目就是只有一个摄像头，双目就是两个摄像头，不过是固定距离的，深度就是不仅能采集到彩色相片，还可以读出每个像素点与相机的距离。除此之外还有全景相机和event相机等新型相机。全景相机就是360度全方位，event相机也叫动态视觉传感器，就是静态时，没有输出，只有当像素点变化超过阈值时，才会有输出，很有意思，但是好像成本和分辨率的问题，应用不如rgbd等其他相机。

### 单目相机

单目SLAM（monocular SLAM）结构简单，成本低，所以比较受关注。照片的本质是拍照时的场景在相机平面的一个投影，他以二维的形式反映了三维的世界。显然，这个过程丢失了一个维度，也就是所谓的深度信息，这样我们就无法得到物体的**距离**。这个距离是一个非常关键的问题。   
我们如果想恢复之前的三维结构，就必须改变相机的视角，比如左右平移和前后远近地移动，这样就能根据视差定量地判断哪些物体离得近，哪些离得远。   
即使我们已经定量地判断出了哪个近哪个远，我们还是不知道具体的距离是多少（因为我们不知道我们左右前后移动了多少，结合其他传感器是一种方法）。换一个说法，如果把相机的运功和相机看到的场景都放大两倍，那么我们将在摄像头看到一模一样的东西。也就是说相对尺度我们可以测定，但是真实尺度我们是无法得知的。   

单目的问题就是：

* 1，只有在移动之后才可以计算深度
* 2，无法确定真实的尺度

### 双目和深度相机

使用这两种的目的就在于，通过某种手段测量物体与我们之间的距离，克服单目相机无法知道距离的弱点。   

* 双目 

双目摄像机是由两个单目组成，但是这两个相机之间的距离（称为基线）是已知的，我们通过基线来估计每个像素的距离（与人眼相似）。基线越大，测量的距离就会越远。   
缺点是配置和标定比较复杂；深度量程和精度受基线和分别率限制；视差的计算非常消耗资源，需要GPU加速或者FPGA才能实时输出距离信息；

* 深度

深度相机是2010年开始兴起的一种相机，最大的特点是通过红外结构光或者tof原理，像极光传感器一样向目标主动发射光并接受，直接测出距离。这与之前的都不一样，可以节省大量计算资源。   
缺点是测量范围窄，噪声大，视野小，易受日光干扰，无法测量透射材质等诸多问题；在SLAM方面，主要用于室内。

# 2.2 经典视觉SLAM框架

通过摄像头我们得到了连续变化的图像，视觉SLAM的目标就是通过这些图像进行定位与建图，这并不简单，并不是通过一个算法就能直接搞定。通过长期的努力，SLAM已经有了一个比较定型的框架了。

![照片](/assets/img/documentation/slam-2-1.png)

整个视觉SLAM流程包括以下步骤。

* 1,传感器信息读取。在视觉SLAM中主要为相机图像信息的读取和预处理。如果是在机器人中，还可能有码盘、惯性传感器等信息的读取和同步。

* 2,视觉里程计(Visual Odometry，VO)。视觉里程计的任务是估算相邻图像间相机的运动，以及局部地图的样子。VO又称为前端。

* 3,后端优化(Optimization)。后端接受不同时刻视觉里程计测量的相机位姿，以及回环检测的信息，对它们进行优化，得到全局一致的轨迹和地图，由于接在VO之后，又称后端。

* 4,回环检测(Loop Closing)。判断机器人是否到达过先前的位置。如果检测到回环，它会把信息提供给后端进行处理。

* 5,建图(Mapping)。根据估计的轨迹，建立与任务要求对应的地图。

后面都会有详细的介绍，这里就先不介绍了，大概就是视觉传感器不靠谱，需要通过回环检测和后端优化进行修正。

### 2.4.2 hello slam

在linux中，程序是一个具有执行权限的文件，可以是脚本也可以是二进制文件，我们一般不限定它的后缀名。下面我们将一个c++程序，使用编译器编译成可以执行的程序。

```
#include <iostream>

int main(int argc, char** argv)
{
    std::cout<<"Hello SLAM!"<<std::endl;
    return 0;
}
```

保存成一个helloslam.cpp文件，然后在它的文件夹下执行：

```
g++ helloslam.cpp
```

这个会生成一个a.out的文件，打开终端，执行这个脚本

```
./a.out
```

就会看见输出Hello SLAM!

题外话：argc是参数的数量，argv是参数，比如`a.out aa bb`argc就是3,argv[1]就是aa

**问题**:一个文件我们可以使用g++，那一个大工程，有无数个文件，我们总不能一个一个地搞吧，于是cmake应运而生。

### 2.4.3 使用cmake

历史上曾经使用makefile进行工程的管理，但是cmake更加方便，我们将使用cmake产生makefile文件，然后再用make编译整个工程。

在刚才的文件夹下新建一个CMakeList.txt，写进去

```
# 声明要求的 cmake 最低版本
cmake_minimum_required( VERSION 2.8 )

# 声明一个 cmake 工程
project( HelloSLAM )

# 设置编译模式
set( CMAKE_BUILD_TYPE "Debug" )

# 添加一个可执行程序
# 语法：add_executable( 程序名 源代码文件 ）
add_executable( helloSLAM helloSLAM.cpp )

# 添加一个库
add_library( hello libHelloSLAM.cpp )
# 共享库
add_library( hello_shared SHARED libHelloSLAM.cpp )

add_executable( useHello useHello.cpp )
# 将库文件链接到可执行程序上
target_link_libraries( useHello hello_shared )
```

**要点**:

* 添加可执行程序格式
* 添加库文件（动态库与静态库）。静态库是在链接阶段将库文件与目标文件一起打包到可执行文件里面，是在编译时期完成的;在运行时与函数库再无瓜葛，移植方便;浪费空间。动态库则是为了改善静态库的问题，首先动态库只有一个副本，在程序编译时不会放进程序，而是在运行时直接调用，所以第一点是节省空间;第二个问题，程序的更新，静态库需要重新将工程编译一遍，动态库则只需要更新库文件;可以实现多程序之间的资源共享。
* 对于库文件的使用者，只需要拿到了头文件和库文件，就可以调用这个库了。
* 在cmake文件中要添加可执行文件，添加库，然后把库文件链接到可执行程序上。
